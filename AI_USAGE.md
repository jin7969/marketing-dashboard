## AI 활용 철학

본 프로젝트에서 AI는 **빠른 코드 생성 도구**가 아니라 **설계 파트너**로 활용했습니다.

AI가 생성한 코드는 반드시 다음 세 가지를 기준으로 검증했습니다.

1. **요구사항 부합 여부** — 과제 명세의 의도를 정확히 반영했는가
2. **기술적 정합성** — 기존 코드베이스의 패턴·타입과 일치하는가
3. **실제 동작 검증** — 브라우저에서 직접 확인했을 때 의도대로 작동하는가

AI 결과물을 그대로 사용한 경우는 거의 없었습니다. 대부분 수정이 필요했고, 그 과정에서 나의 판단을 우선시했습니다.

---

## 컨텍스트 설계 전략

단순히 "만들어줘" 방식이 아니라, AI가 올바른 판단을 할 수 있도록 **충분한 컨텍스트를 먼저 구성**했습니다.

- 새 기능 구현 시 관련 파일(타입, 스토어, 유틸)을 함께 제공해 기존 패턴과 일관된 코드를 유도
- 추상적인 요청 대신 **제약 조건을 명시한 구체적인 프롬프트** 사용
- 복잡한 기능은 한 번에 요청하지 않고, 설계 → 검증 → 구현 순서로 단계를 나눠 진행

## 주요 활용 사례 및 검증 과정

### 1. 아키텍처 설계 — AI 제안 수용 + 일부 기각

**프롬프트:**

> "글로벌 필터가 모든 차트·테이블에 연동되고, 캠페인 등록 후 즉시 반영되어야 합니다. React + TypeScript + Zustand + TanStack Query 조합으로 폴더 구조를 설계해주세요."

**AI 제안:** `api / components / hooks / store / utils` 레이어 구조 + `context/` 폴더 별도 구성

**내 판단 및 수정:**
Zustand를 이미 도입한 상황에서 Context API를 병행하면 구독 방식이 이중화되어 불필요한 복잡도가 생긴다고 판단했습니다. `context/` 폴더를 제거하고 모든 전역 상태를 Zustand로 일원화했습니다.

---

### 2. 날짜 포맷 불일치 — AI 오류를 직접 발견

**프롬프트:**

> "date-fns로 당월 1일~말일을 계산해서 Zustand 초기값으로 설정해주세요."

**AI 제안:** `format(date, 'yyyy/MM/dd')` 형식의 문자열로 날짜를 생성해 비교

**내 판단 및 수정:**
직접 `db.json`을 확인한 결과 날짜 필드가 `YYYY-MM-DD`(하이픈) 형식이었습니다. 최종 구현에서는 `format()` 대신 `startOfMonth()/endOfMonth()`로 Date 객체를 직접 사용하고, db.json 날짜 파싱 시 `replace(/\//g, '-')`로 방어적으로 처리해 슬래시·하이픈 혼용 상황을 모두 대응했습니다. AI는 db.json을 직접 보지 않았기 때문에 포맷 불일치를 스스로 잡을 수 없었습니다.

---

### 3. 플랫폼 도넛 차트 — 요구사항 해석 차이 수정

**프롬프트:**

> "도넛 차트에서 플랫폼을 클릭하면 글로벌 필터에 연동되어야 합니다. 단, 모든 플랫폼이 해제되는 건 막아야 합니다."

**AI 제안:** 선택된 플랫폼의 데이터만 도넛에 표시하는 방식으로 구현

**내 판단 및 수정:**
과제 명세를 다시 읽어보니 요구사항은 "차트는 전체 플랫폼 비중을 항상 보여주되, 클릭이 글로벌 필터에만 반영"되는 것이었습니다. AI는 "클릭 = 차트 필터링"으로 해석했지만, 실제 의도는 "클릭 = 상단 필터 연동"이었습니다. 도넛 데이터 계산에서 플랫폼 필터를 제외하고 날짜·상태 필터만 적용하도록 수정했습니다.

---

### 4. 일괄 상태 변경 — AI 결과물에 UX 관점 추가

**상황:** 캠페인을 체크박스로 선택하고 상태를 일괄 변경한 후, 드롭다운이 이전 선택값으로 남아있는 버그

**AI 제안:** `onSuccess`에서 드롭다운 상태만 초기화

**내 판단 및 수정:**
드롭다운만 리셋하면 체크박스는 선택된 상태로 남습니다. "변경이 완료됐는데 아직 선택 중"인 것처럼 보이는 UX가 부자연스럽다고 판단해, `selectedIds`(체크박스 선택 상태)도 함께 초기화했습니다. AI는 버그 수정 범위만 다뤘지만, 저는 사용자 흐름 전체를 고려했습니다.

---

### 5. 필터 변경 시 페이지 리셋 — 엣지 케이스 직접 추가

**프롬프트:**

> "글로벌 필터가 변경될 때 테이블 페이지를 1로 리셋해주세요."

**AI 제안:** `dateRange`, `statuses`, `platforms`의 변경을 감지해 페이지 리셋

**내 판단 및 수정:**
직접 테스트하다가 검색어(`searchTerm`)를 바꿔도 현재 페이지가 유지되어 빈 페이지가 나타나는 동일한 문제를 발견했습니다. AI는 명시적으로 요청한 조건만 처리했지만, 같은 문제가 발생하는 시나리오를 직접 찾아내 `filterKey`에 `searchTerm`을 포함시켜 검색어 변경 시에도 페이지가 리셋되도록 추가했습니다.

```ts
const filterKey = `${statuses.join()}-${platforms.join()}-${dateRange.startDate.getTime()}-${dateRange.endDate.getTime()}-${searchTerm}`;
```

---

### 6. `useEffect` vs 파생 상태 — AI 패턴 기각

**상황:** 페이지 리셋 구현 시

**AI 제안:** `useEffect`로 필터 의존성을 감지해 `setCurrentPage(1)` 호출

**내 판단 및 수정:**
`useEffect`로 상태를 업데이트하면 렌더가 두 번 발생(필터 변경 → 렌더 → 리셋 → 렌더)합니다. `filterKey`를 계산하고 현재 페이지를 파생 상태로 처리하는 방식으로 변경해 불필요한 이중 렌더를 제거했습니다.

```ts
// AI 제안 — useEffect 방식 (렌더 2회)
useEffect(() => {
  setCurrentPage(1);
}, [statuses, platforms, dateRange]);

// 내 수정 — 파생 상태 방식 (렌더 1회)
const filterKey = `${statuses.join()}-${platforms.join()}-${dateRange.startDate.getTime()}-...`;
const [pageState, setPageState] = useState({ filterKey, page: 1 });
const currentPage = pageState.filterKey === filterKey ? pageState.page : 1;
```

## AI가 잘한 것 / 내가 판단한 것

| 영역           | AI가 기여한 부분                             | 내가 직접 판단한 부분                 |
| -------------- | -------------------------------------------- | ------------------------------------- |
| 코드 생성 속도 | 반복적인 컴포넌트 구조, 타입 정의, 포맷 함수 | 설계 방향, 레이어 구분 기준           |
| 패턴 제안      | 훅 분리, React Query 사용법                  | 어떤 상태를 어디에 둘지 최종 결정     |
| 버그 수정      | 증상에 대한 수정 코드 생성                   | 근본 원인 파악, 관련 엣지 케이스 발굴 |
| 요구사항 구현  | 명시된 기능 구현                             | 명세에 없는 UX 개선, 암묵적 요구 판단 |

AI는 구현 속도를 크게 높였지만, **무엇을 만들지, 왜 그렇게 만드는지는 항상 내가 먼저 결정**하고 AI에게 구체적인 방향을 제시했습니다.
